# YouTube Video Fact-Checker — Architecture Document

**Version:** 1.1
**Date:** 2026-03-29
**Status:** Approved (PRDE Phase 3 — Design)
**Supersedes:** v1.0 (2026-03-27)
**Change summary:** Added multi-provider LLM support (Gemini + Anthropic), three-tier model routing, per-stage provider configuration. No changes to Core interfaces, domain models, pipeline, scoring, API, or UI.

---

## 1. Overview

A web application that accepts a YouTube URL and produces a structured summary, per-claim fact-checks with source citations, and a watch-worthiness assessment. Designed for friends-scale usage (single instance, no multi-tenancy). Stateless — each analysis is independent with no persistence.

### 1.1 Core Workflow

```
User pastes URL → Extract transcript → Detect domain → Summarise + Extract claims (parallel)
→ Verify each claim with web search (parallel) → Validate source URLs → Score → Assess
→ Stream progressive results to user
```

### 1.2 Key Design Decisions

| Decision | Rationale |
|---|---|
| API-first architecture | Enables future mobile clients consuming the same JSON API |
| Thin Razor + HTMX web layer | Minimal frontend complexity; disposable when a richer client is built |
| Multi-provider LLM support | Switch between Anthropic and Gemini (or future providers) via config; cost and quality flexibility |
| Three-tier model routing | Fast / Standard / Premium per stage; right-size model cost to task complexity |
| Per-stage model tier assignment | Configurable via `StageModelOptions`; tune without code changes |
| Deterministic scoring (not LLM) | Transparency and reproducibility (Rule R7) |
| Channel-based event transport | In-process, no external dependencies; sufficient for single-instance |
| Pluggable transcript extraction | De-risks YouTube API changes; swap implementations without touching pipeline |

### 1.3 Technology Stack

| Component | Technology | Rationale |
|---|---|---|
| Backend | C# / ASP.NET Core 9 | Team strength; strong async/pipeline support |
| Web UI (v1) | Razor Pages + HTMX + Pico CSS | Thin disposable layer; no JS build pipeline |
| LLM Provider (default) | Google Gemini API (2.5 Flash, 3 Flash, 2.5 Pro) | 10-20x cheaper than Anthropic; Google Search grounding for fact verification |
| LLM Provider (alternate) | Anthropic Claude API (Sonnet + Haiku) | Higher reasoning quality; fallback if Gemini quality insufficient |
| Transcript Extraction | YoutubeExplode (NuGet) | Mature .NET library for YouTube data |
| Deployment | Docker, single container | Simple; single host for friends-scale |


## 5. Analysis Events

Events are published by the pipeline and consumed by the SSE endpoint. Each event maps to a UI update.

```csharp
public abstract record AnalysisEvent(string AnalysisId, DateTimeOffset Timestamp);

public record AnalysisStartedEvent(...)      : AnalysisEvent  // VideoInfo
public record TranscriptExtractedEvent(...)  : AnalysisEvent  // Quality, WordCount
public record DomainDetectedEvent(...)       : AnalysisEvent  // ContentDomain
public record SummaryCompleteEvent(...)      : AnalysisEvent  // Summary
public record ClaimsExtractedEvent(...)      : AnalysisEvent  // List<Claim>
public record ClaimVerifiedEvent(...)        : AnalysisEvent  // FactCheck (per-claim)
public record ScoringCompleteEvent(...)      : AnalysisEvent  // ScoreBreakdown
public record AssessmentCompleteEvent(...)   : AnalysisEvent  // Assessment
public record AnalysisFailedEvent(...)       : AnalysisEvent  // AnalysisError
```

### 5.1 SSE Event Format

```
event: analysis.started
data: {"analysisId":"...","video":{"title":"...","channel":"...","duration":"00:14:32"}}

event: claim.verified
data: {"analysisId":"...","factCheck":{"claimId":"claim-03","verdict":"Refuted",...}}

event: analysis.failed
data: {"analysisId":"...","error":{"stage":"TranscriptExtraction","message":"...","recoverable":false}}
```

### 5.2 Event Transport

`System.Threading.Channels.Channel<AnalysisEvent>` — one unbounded channel per active analysis. The pipeline writes to the channel; the SSE endpoint reads from it. In-process only; no external message broker needed at friends-scale.

## 6. Pipeline Architecture

### 6.1 Execution Flow

```
URL → Validate → Extract Transcript → Detect Domain ─┬─→ Summarise ──────────────┐
                                                      │                            │
                                                      └─→ Extract Claims ──────────┤
                                                                                   │
                                           ┌───────────────────────────────────────┘
                                           │
                                           ▼
                                    Verify Claims (parallel, bounded concurrency)
                                           │
                                           ▼
                                    Validate Source URLs
                                           │
                                           ▼
                                    Score (deterministic)
                                           │
                                           ▼
                                    Assess (LLM synthesis)
```

### 6.2 Parallelism

- **Summarisation + Claim Extraction** run in parallel after domain detection (neither depends on the other).
- **Claim Verification** runs up to `MaxConcurrentVerifications` (default 4) claims in parallel via `Parallel.ForEachAsync`.
- **Source URL Validation** runs in parallel per-claim (lightweight HTTP HEAD requests).

### 6.3 Pipeline Timeout

Global pipeline timeout: `PipelineTimeoutSeconds` (default 120). Enforced via `CancellationTokenSource` linked to the caller's token.

### 7.1 Transcript Extraction

### 7.2 LLM Integration — Multi-Provider Architecture

The LLM layer is structured in three tiers: a **provider client abstraction** (`ILlmClient`), **provider-specific implementations** (Gemini, Anthropic), and **provider-agnostic stage implementations** that contain prompts and response parsing.

```
Infrastructure/
  LlmProviders/
    Common/
      ILlmClient.cs                 ← Provider-agnostic client interface
      LlmRequest.cs                 ← Common request type
      LlmResponse.cs                ← Common response types
      LlmSearchResponse.cs
      SearchResultSource.cs
      TokenUsage.cs
      StructuredOutputParser.cs      ← Shared JSON parsing utility
    Anthropic/
      AnthropicLlmClient.cs         ← ILlmClient implementation
      AnthropicWebSearchParser.cs    ← Extracts Sources from tool_use responses
    Gemini/
      GeminiLlmClient.cs            ← ILlmClient implementation
      GeminiOptions.cs
      GeminiGroundingParser.cs       ← Extracts Sources from groundingMetadata
    Stages/
      DomainDetectorStage.cs         ← IDomainDetector (uses ILlmClient)
      SummariserStage.cs             ← ISummariser (uses ILlmClient)
      ClaimExtractorStage.cs         ← IClaimExtractor (uses ILlmClient)
      ClaimVerifierStage.cs          ← IClaimVerifier (uses ILlmClient.CompleteWithSearchAsync)
      AssessmentGeneratorStage.cs    ← IAssessmentGenerator (uses ILlmClient)
    ServiceCollectionExtensions.cs   ← AddLlmProvider() DI registration
```

Note: `ModelTier` enum lives in `Core/Enums/ModelTier.cs` (not in LlmProviders). `AnthropicOptions` lives in `Infrastructure/Options/`. `GeminiOptions` lives in `LlmProviders/Gemini/`.

#### 7.2.2 Provider Implementations

**Gemini (`GeminiLlmClient`):**
- Maps model tiers to configurable model strings (default: Fast → `gemini-2.5-flash`, Standard → `gemini-3-flash`, Premium → `gemini-2.5-pro`)
- `CompleteAsync` sends `generateContent` requests to the Gemini API
- `CompleteWithSearchAsync` enables Google Search grounding via `tools: [{ google_search: {} }]`
- `GeminiGroundingParser` extracts `SearchResultSource` records from `groundingMetadata.groundingChunks` and maps `groundingSupports` segments to snippets
- Retry with exponential backoff on 429/500/503
- Structured JSON output parsing (handles markdown fences, whitespace)

**Anthropic (`AnthropicLlmClient`):**
- Maps model tiers to configurable model strings (default: Fast → `claude-haiku-4-5-20251001`, Standard → `claude-sonnet-4-20250514`, Premium → `claude-sonnet-4-20250514`)
- `CompleteAsync` sends standard messages API requests
- `CompleteWithSearchAsync` enables web search via `tools: [{ type: "web_search_20250305", name: "web_search" }]`
- `AnthropicWebSearchParser` extracts `SearchResultSource` records from interleaved `tool_use`/`tool_result` content blocks
- Retry with exponential backoff on 429/500/503

#### 7.2.3 Stage Implementations (Provider-Agnostic)

Each stage implements a Core interface and depends only on `ILlmClient`. The stage owns the **prompt** and the **response parsing**. The client owns the **transport**.

Prompts are identical across providers — both Gemini and Anthropic consume the same natural language instructions and return JSON. Provider differences (how search is triggered, how results are structured) are fully encapsulated in the `ILlmClient` implementations.

Each stage reads its model tier from `StageModelOptions` (configurable, not hard-coded).

## 8. API Layer

### 8.1 Endpoints

```
POST /api/analyse
  Request:  { "url": "https://youtube.com/watch?v=..." }
  Response: { "analysisId": "guid" }
  Status:   202 Accepted
  Notes:    Validates URL format, starts pipeline on background task

GET /api/analyse/{id}/stream
  Response: text/event-stream (SSE)
  Events:   JSON-serialised AnalysisEvent subtypes
  Notes:    Client receives events as pipeline progresses; stream closes on completion or failure

GET /api/analyse/{id}
  Response: Complete AnalysisResult as JSON
  Status:   200 if complete, 202 if running, 404 if unknown
  Notes:    For polling or late-joining clients
```

### 8.2 Lifecycle Management

- In-memory `ConcurrentDictionary<string, AnalysisResult>` keyed by analysis ID.
- Pipeline runs on `Task.Run` after 202 response.
- No persistence — results lost on restart. Acceptable for v1.
- Architecture does not block adding persistence later (introduce `IAnalysisRepository` backed by a database).

## 10. Scoring Algorithm

Scoring is deterministic domain logic (Rule R7 — LLMs do not score their own work).

### 10.1 Accuracy Score (0–100)

```
For each claim (excluding Verdict.NotAClaim):
  weight = claim.Importance / sum(all claim importances)
  claimScore = verdict switch {
    Supported          → 100,
    PartiallySupported → 60,
    Unverifiable       → 40,
    Refuted            → 0
  }
  weightedScore += weight * claimScore

// Rule R4: Penalise worst claims
worstClaimPenalty = count(Refuted claims with Importance >= 4) * 15, capped at 30

AccuracyScore = max(0, weightedScore - worstClaimPenalty)
```

### 10.2 Source Quality Score (0–100)

```
accessibleRatio = count(IsAccessible sources) / count(all sources)
SourceQualityScore = accessibleRatio * 100
```

### 10.3 Verifiability Score (0–100)

```
verifiableRatio = count(claims where Verdict != Unverifiable) / count(all claims)
VerifiabilityScore = verifiableRatio * 100
```

### 10.4 Aggregate Score (0–100)

```
AggregateScore =
  0.60 * AccuracyScore +
  0.20 * SourceQualityScore +
  0.15 * VerifiabilityScore +
  0.05 * transcriptQualityBonus    // +5 if manual captions, 0 otherwise
```

Weights are configurable. The `ScoreMethod` string in the output identifies the algorithm version for transparency.

### 10.5 Scoring as Optional Stage

If scoring proves problematic (PM6 risk), the pipeline can skip scoring and assessment entirely. All upstream outputs (summary, claims, verdicts) still deliver value independently.

## 12. Error Handling Strategy

| Stage | Failure Mode | Behaviour |
|---|---|---|
| URL validation | Invalid URL, non-YouTube | Fail fast, 400 response |
| Metadata fetch | Video not found / private | Fail, publish `AnalysisFailed` |
| Transcript extraction | No captions available | Fail, clear user message |
| Domain detection | LLM error / timeout | Default to `General`, continue |
| Summarisation | LLM error / timeout | Fail pipeline |
| Claim extraction | LLM error / timeout | Fail pipeline |
| Claim verification (individual) | LLM error on one claim | Mark `Unverifiable`, continue |
| Source validation | HTTP timeout | Mark `IsAccessible = false`, continue |
| Scoring | Logic error | Fail pipeline (shouldn't happen) |
| Assessment | LLM error | Skip, deliver results without assessment |

**Principle:** Fail hard on core-value stages (transcript, claims, verification). Degrade gracefully on enrichment stages (domain detection, assessment, individual sources).

## 14. Rules and Invariants

| # | Rule | Enforcement |
|---|---|---|
| R1 | Claims must be falsifiable factual assertions — not opinions | Prompt engineering + `NotAClaim` verdict |
| R2 | Every FactCheck must cite ≥1 source or state "no reliable source found" | Prompt constraint + validation in parser |
| R3 | Claims reclassified as Opinion excluded from scoring | `NotAClaim` verdict filtered in scoring engine |
| R4 | Aggregate score penalises worst claims, not just average | `worstClaimPenalty` in scoring algorithm |
| R5 | No transcript → graceful failure with clear message | Pipeline error handling |
| R6 | Assessment distinguishes "accurate but low-density" from "dense but unreliable" | Assessment prompt design |
| R7 | Scoring is deterministic domain logic, not LLM-generated | `IScoringEngine` is pure computation |
